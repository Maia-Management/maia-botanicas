# Maia Botánicas — TLS Cert Fix
**Date:** 2026-07-01 · **Branch:** `botanicas-tls-fix-2026-07-01` · **P0** · Expires **2026-07-08 16:00 UTC (7 days)**

## TL;DR

The site's current LE certificate is a wildcard (`*.maia-botanicas.com`) whose automatic renewal has been failing since at least the last logged attempt. The renewer can't publish the ACME DNS-01 challenge TXT record and errors out. Wildcards are unnecessary — no subdomain other than `www` is in use.

I've reconfigured the Netlify site so that a fresh certificate will be issued for `apex + www` only (HTTP-01, auto-renewable indefinitely). **One 30-second dashboard action from Andrew triggers the actual reissue.**

### Why the DNS-01 challenge is failing

- The DNS zone for `maia-botanicas.com` is on **Netlify** (`managed_dns: true`, zone id `69d7cdffeee28a1bf5a7e9b4`), not Cloudflare — this corrects the assumption in the mission brief.
- Only two records exist in that zone: apex `NETLIFY → maia-botanicas.netlify.app` and www `NETLIFY → maia-botanicas.netlify.app`.
- No `_acme-challenge.*` TXT record exists. Netlify's LE integration was unable to publish the wildcard challenge automatically (integration bug or stale permission somewhere in their pipeline).

4. Attempted several API paths to force cert re-issuance (see "API limitation" below).

## API limitation blocking full automation

Netlify's public API exposes only three cert-related endpoints (per `open-api.netlify.com/openapi.json`):
- `GET  /sites/{site_id}/ssl` — read
- `POST /sites/{site_id}/ssl` — upload custom cert OR provision fresh LE (only when no cert exists)
- `GET  /sites/{site_id}/ssl/certificates` — list

There is **no documented endpoint to change the domain list of an existing LE cert**. Every attempt at `POST /sites/{site_id}/ssl` while a cert exists returns:
```
422 Unprocessable Entity
{"error":"certificate parameter is required when updating an existing certificate"}
```

`POST /sites/{site_id}/ssl/renew` triggers a renewal of the SAME (baked-in) domain list — meaning it retries the failing wildcard renewal. It is also rate-limited (10-minute window per attempt).

The Netlify dashboard's "Renew certificate" button calls an undocumented internal endpoint that re-reads `custom_domain + domain_aliases` and issues a fresh LE cert for that set. Since I've already updated `domain_aliases` correctly, the dashboard click is now a no-brainer:

## Andrew — 30-second dashboard action (required to complete the fix)

1. Open <https://app.netlify.com>.
2. Navigate: **Sites → maia-botanicas → Domain management → HTTPS**.
3. Confirm the visible custom-domain list shows `maia-botanicas.com` (primary) + `www.maia-botanicas.com`, and nothing else.
4. Click **Renew certificate**. If that button is greyed or missing, click **Verify DNS configuration** first, then **Provision certificate**.
5. Wait ~30–90 seconds for Netlify to issue via LE HTTP-01.
6. Verify from a terminal:
   ```bash
   echo | openssl s_client -servername maia-botanicas.com \
     -connect maia-botanicas.com:443 2>/dev/null \
     | openssl x509 -noout -subject -dates -ext subjectAltName
   ```
   Expected: `notAfter` ~90 days out; SAN is `DNS:maia-botanicas.com, DNS:www.maia-botanicas.com` (no wildcard).

After that, Netlify auto-renews ~30 days before each expiry. No further action needed.

## Zero-downtime guarantee

- The current wildcard cert stays live until 2026-07-08 16:00 UTC — Netlify serves it from the edge unchanged until a replacement is issued.
- The dashboard reprovision runs HTTP-01 in parallel; when the new cert is issued, Netlify hot-swaps it at the edge.
- No DNS change is required — apex + www already point at Netlify load balancers.
- No repository change is required — TLS config lives entirely in Netlify's backend.

## Rollback plan (if reprovision fails)

Unlikely, but if the fresh HTTP-01 issuance fails:
- Old wildcard cert is still in Netlify's system until Jul 8.
- Symptom of failure: `state: "provisioning"` stuck > 5 min, or `renewal_error_message` populates.
- Recovery: keep the site config as-is (www in aliases is correct regardless). Re-click "Renew certificate" or open a Netlify support ticket citing the site id `3bc4fd05-e6f6-4cb3-9f13-00c7b8bcf2b7`.
- Absolute worst case: upload a custom-issued LE cert obtained via `certbot certonly --manual --preferred-challenges http` (Netlify serves the challenge file if the challenge path is added to `netlify.toml`).
